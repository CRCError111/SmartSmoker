Полная система управления умной коптильней с облачной интеграцией
1. ОБЩАЯ АРХИТЕКТУРА СИСТЕМЫ
1.1. Компоненты системы
Система состоит из трёх взаимосвязанных компонентов:
Аппаратная часть (ESP32):
	ESP32-WROOM-32 с Wi-Fi
	Датчики: BME280 (температура/влажность камеры), 2×NTC (температура дыма/продукта)
	Исполнительные механизмы: 
		SSR (управление ТЭН для нагрева камеры), 
		3 MOSFET (управление вентилятором дымогенератора, управление вентилятором внутри камеры, управление вентилятором для провертривания), 
		сервопривод заслонки.
	OLED-дисплей 2.42" SSD1306
	4 тактовые кнопки (MENU, UP, DOWN, OK)
Облачный сервис (Веб-сайт):
	PHP 8.0+ с фреймворком на основе Tabler UI
	MySQL 5.7+ база данных
	RESTful API для взаимодействия с ESP32
	Реактивный веб-интерфейс с графиками и управлением
	Мобильное приложение (в перспективе):
	React Native приложение
	Push-уведомления о завершении программ
	Offline-режим управления

1.3. Сценарии использования
	Сценарий 1: Первоначальная настройка
		1.1 Пользователь включает ESP32 → устройство переходит в режим AP, пользователь подключается к точке доступа SmartSmoker_AP_XXXX, в браузере открывается главная страница с возможностью настройки подключения к домашней сети Wi-Fi, пользователь вводит реквизиты домашней сети,	ESP32 перезагружается и подключается к указанной сети.
		1.2 Если пользователь не вводит реквизиты для подключения к домашней сети WiFi, то контроллер ESP32 работает автономно - в веб-интерфейсе отображает показания датчиков, позволяет через веб-интерфейс создавать и редактировать программы, запускать их, то есть используется весь локальный функционал, кроме отображения данных на сайте.
		1.3 Если есть необходимость для пользователя - то он вводит логин и пароль Веб-сайта в отдельной странице, после чего пользователь заходит на сайт, создает на сайте устройство и привязывает его к ESP32 через Device ID. После этого пользователь получает возможность видеть показания датчиков онлайн, кроме того будет сохраняться на Веб-сайте история показаний за последние несколько запусков.
	Сценарий 2: Повседневное использование
		2.1 Пользователь заходит на сайт → автоматическая авторизация (токен в localStorage). На главной странице видит список устройств с мини-графиками последних запусков. Выбирает устройство → видит детальные графики с историей последних 10 запусков программ. Каждый запуск идентифицируется уникальным run_id, доступна детализация по конкретному запуску.
		2.2 Нажимает "Запустить программу" → выбирает программу из списка, либо создает новую по своим параметрам или из шаблона. Если программа новая — программа отправляется на ESP32 с уникальным Device ID. ESP32 обязан подтвердить получение программы (POST /api/program-received.php).
		2.3 ESP32 начинает выполнение программы. Пользователь отслеживает процесс через графики и получает уведомления по email. При необходимости нажимает "Аварийная остановка" → немедленная остановка всех процессов и отправка email-уведомления.
	Сценарий 3: Автономное управление
		3.1 Потеряна связь с облаком, нет интернета либо пользователь использует устройство автономно. Пользователь управляет коптильней через OLED-дисплей и кнопки:
			MENU → переход в меню программ
			UP/DOWN → выбор программы
			OK → запуск выбранной программы
			Долгое MENU (3 сек) → аварийная остановка
		3.2 Если проблема была в подключении к сайту, потере интернета - то при восстановлении связи данные синхронизируются с облаком, но используются только с уникальным Device ID, иначе устройство продолжает функционировать автономно.

	ВАЖНО: Двухфайловая система хранения состояния привязки:
		- binding.json (BindingManager) - источник истины, содержит состояние привязки
		- cloud.json (StorageManager) - синхронизируется с binding.json при запуске
		- При запуске ESP32 выполняет reconciliation (согласование) между файлами
		- binding.json имеет приоритет при конфликтах
		- Оба файла обновляются атомарно при операциях bind/unbind

	Старый формат (плоская структура) - поддерживается для обратной совместимости:
	{
	  "program_name": "Название программы",
	  "description": "Описание",
	  "category": "custom",
	  "stages": [...]
	}
	
	Парсер программ (ProgramManager::parseProgramFromJson) автоматически определяет формат
	и корректно обрабатывает оба варианта. При сохранении используется новый формат.
	
	Формы создания и редактирования программ на контроллере содержат ВСЕ поля,
	доступные на сайте, что гарантирует полную совместимость:
	- Все параметры этапов (включая hysteresis, ventilation_percent, compressor_pwm)
	- Программы с сайта полностью редактируются на контроллере
	- Программы с контроллера полностью редактируются на сайте
	- Все параметры сохраняются при передаче между системами
2.3. Функциональные требования
2.3.1. Сбор и обработка данных
	Частота опроса датчиков: каждые 2 секунды
	Фильтрация данных:
		АЦП: среднее из 8 измерений с задержкой 2 мс между измерениями
	Проверка на валидность диапазонов:
		NTC: -50°C...200°C, напряжение 0.01V...3.62V
		BME280: -40°C...85°C, влажность 0%...100%
	Обработка ошибок датчиков:
		При последовательной ошибке датчика более трёх раз устанавливается флаг sensorError = true
		Система переходит в режим EMERGENCY_STOP
		Все исполнительные механизмы отключаются
		На OLED отображается сообщение об ошибке
2.3.2. Управление исполнительными механизмами
	Терморегуляция:
		Гистерезис: ±2°C от целевой температуры
		Минимальное время между переключениями ТЭНа: 5 секунд
		Аварийная остановка при достижении 100°C
		Управление заслонкой:
			Процентное открытие: 0% (закрыто) - 100% (полностью открыто)
			Угол поворота: 0° - 90° (линейное преобразование)
			Плавное движение: шаг 5° с задержкой 100 мс
	Вентиляторы:
		Вентилятор в камере: включение при превышении температуры
		Вентилятор подачи воздуха: управление влажностью на этапах сушки
	Дымогенератор:
		ШИМ управление (0-100% мощности)
		Автоматический режим при compressorPWM = -1
		Ручной режим с указанием мощности
2.3.3. Локальный интерфейс (OLED + кнопки)
	Режимы отображения:
		MAIN_SCREEN - Основной экран с параметрами
		PROGRAM_SELECTION - Выбор программы из списка
		PROGRAM_RUNNING - Выполнение программы
		SETTINGS_MENU - Меню настроек
		EMERGENCY_STOP - Экран аварийной остановки
	Логика кнопок:
		Короткое нажатие (<1 сек): основное действие
		Долгое нажатие (>1 сек): дополнительное действие
		Двойное нажатие: быстрый доступ к часто используемым функциям
	Антидребезг:
		Программная фильтрация: 50 мс после изменения состояния
		Аппаратная фильтрация: конденсаторы 100 нФ на линиях кнопок
	Энергосбережение:
		Автоматическое отключение подсветки OLED через 30 сек бездействия
		Яркость снижается до 30% за 5 сек перед полным отключением
		Восстановление яркости при любом взаимодействии

2.4.3. Безопасность взаимодействия
	Аутентификация:
		Все запросы ESP32 к облаку содержат device_id и api_token
		api_token выдаётся сервером при первичной привязке (POST /api/bind-device.php)
		Проверка существования устройства в базе перед приемом данных
		Неверный device_id или невалидный api_token → HTTP 401 Unauthorized
	Валидация данных:
		Проверка диапазонов значений перед применением
		Максимальный размер JSON: 4 КБ
		Защита от переполнения буфера
	Защита от атак:
		Ограничение скорости запросов: максимум 10 запросов/сек с одного IP
		Таймаут сессии сайта: 5 минут бездействия
2.4.4. Подтверждение получения файлов (новое требование)
	Все файлы, передаваемые с сайта на ESP32, должны подтверждаться:
		- Программы (POST /api/load-from-cloud)
		- Настройки WiFi (POST /api/wifi)
		- Настройки облака (POST /api/cloud-settings)
		- OTA обновления (POST /update)
	ESP32 сохраняет файл в LittleFS и отправляет подтверждение с:
		- "success": true при успешном сохранении
		- "success": false + "error": "сообщение об ошибке" при ошибке
	Сайт считает файл переданным только после получения подтверждения
	При отсутствии подтверждения в течение 10 секунд сайт повторяет отправку (максимум 3 раза)
	После 3 неудачных попыток файл помечается как не переданный и требует ручного вмешательства

4. ДЕТАЛЬНОЕ ОПИСАНИЕ ВЗАИМОДЕЙСТВИЯ ESP32 И САЙТА
4.1. Процесс инициализации и аутентификации
4.1.1. Первичная настройка ESP32 (в режиме AP)
	1. При первом запуске (или сбросе настроек):
		ESP32 автоматически переходит в режим точки доступа (AP)
		Создает сеть с именем SmartSmoker_AP_XXXX (где XXXX - последние 4 символа MAC-адреса)
		Запускает веб-сервер на IP-адресе 192.168.4.1
		Отображает на OLED:
			Smart Smoker SETUP
			Wi-Fi: AP Mode
			SSID: SmartSmoker_AP_1234
			IP: 192.168.4.1
	2. Пользователь подключается к точке доступа:
		Открывает браузер и переходит на 192.168.4.1
		Видит страницу wifi.html с формой для ввода:
		SSID домашней сети
		Пароль к сети
		При отправке формы данные сохраняются в файл /wifi.json:
			{
			  "ssid": "HomeWiFi",
			  "pass": "encrypted_password",
			}
		ESP32 перезагружается и пытается подключиться к указанной сети

4.1.2. Подключение к сети и переход в режим STA
	1. После перезагрузки:
		ESP32 загружает настройки из /wifi.json
		Расшифровывает пароли
		Пытается подключиться к указанной Wi-Fi сети
		При успешном подключении:
		Отображает на OLED:
			Smart Smoker ONLINE
			Wi-Fi: STA Mode
			SSID: HomeWiFi
			IP: 192.168.1.105
		Запускает фоновую проверку связи с облаком
	2. При неудачном подключении:
	Через 15 секунд возвращается в режим AP
	Отображает на OLED:
		Smart Smoker ONLINE
		Wi-Fi: STA Mode
		SSID: HomeWiFi
		IP: 192.168.1.105
	3. При удачном подключении 
		Видит ссылку для подключения устройства к сайту	
		Вводит логин и пароль пользователя сайта
		При отправке формы данные сохраняются в файл /site.json:
			{
				"cloud_login": "user@example.com",
				"cloud_password": "encrypted_cloud_password",
				"cloud_url": "https://crcerror.ru"
			}
		ESP32 пытается подключиться к указанной сети		
		
4.1.3. Создание устройства на сайте и привязка к ESP32
	1. Пользователь заходит на сайт crcerror.ru:
		Авторизуется (admin/password для демо-версии)
		Переходит в раздел "Устройства" → "Добавить устройство"
		Заполняет форму с названием устройства
		Сайт генерирует уникальный Device ID (UUID v4)
		Создает запись в базе со статусом "pending":
			INSERT INTO devices (user_id, name, device_id, status) 
			VALUES (123, 'Коптильня на кухне', 'a1b2c3d4-e5f6-7890-g1h2-i3j4k5l6m7n8', 'pending')
	2. Пользователь переходит на страницу привязки:
		Вводит полученный Device ID и IP-адрес ESP32
		Сайт отправляет POST-запрос на ESP32:	
			POST http://192.168.1.105/api/bind-device
			{
			  "device_id": "a1b2c3d4-e5f6-7890-g1h2-i3j4k5l6m7n8"
			}
	3. ESP32 обрабатывает запрос:
		Проверяет формат Device ID (должен быть UUID v4)
		Проверяет наличие файла /cloud.json в LittleFS
		Если файл существует и содержит device_id:
			Сравнивает device_id с полученным
			Если совпадает - пропускает повторную привязку
			Если не совпадает - удаляет старый файл и продолжает привязку
		Если файл не существует - начинает привязку
		Отправляет запрос привязки на облако:
			POST https://crcerror.ru/api/bind-device.php
			{ "device_id": "a1b2c3d4-...", "esp32_ip": "192.168.1.105", "mac_address": "...", "firmware_version": "2.0.0" }
		Облако возвращает api_token и параметры синхронизации:
			{ "success": true, "device_token": "<api_token>", "device_name": "Коптильня", "sync_interval": 60, "server_time": "..." }
		ESP32 сохраняет в два файла:
			/binding.json (управляется BindingManager):
			{
			  "uuid": "a1b2c3d4-e5f6-7890-g1h2-i3j4k5l6m7n8",
			  "api_token": "<api_token>",
			  "bound": true,
			  "username": "user@example.com",
			  "timestamp": "2026-01-26T12:34:56Z",
			  "website": "https://crcerror.ru"
			}
			/cloud.json (управляется StorageManager):
			{
			  "device_id": "a1b2c3d4-e5f6-7890-g1h2-i3j4k5l6m7n8",
			  "api_token": "<api_token>",
			  "device_name": "Коптильня",
			  "device_bound": true,
			  "cloud_url": "https://crcerror.ru",
			  "sync_interval": 60,
			  "program_sync_interval": 300
			}
		КРИТИЧЕСКИ ВАЖНО: Атомарное обновление состояния
			После успешного сохранения binding.json через BindingManager:
			- Немедленно обновляет systemState.deviceId, systemState.deviceBound, systemState.apiToken
			- Немедленно вызывает storageManager->saveCloudSettings() для сохранения в cloud.json
			- При ошибке сохранения cloud.json выполняет откат (rollback):
				* Вызывает bindingManager->unbind() для удаления binding.json
				* Очищает systemState (deviceBound=false, deviceId="", apiToken="")
				* Возвращает HTTP 500 с сообщением об ошибке
			- После успешного сохранения обоих файлов выполняет верификацию:
				* Сравнивает binding.json (через bindingManager->isBound()) и cloud.json (через systemState)
				* Проверяет согласованность: bound == deviceBound, uuid == deviceId, api_token == apiToken
				* Логирует результат верификации (успех или предупреждение о несоответствии)
		Обновляет глобальное состояние: systemState.deviceId, systemState.apiToken, systemState.deviceBound
		Возвращает сайту подтверждение привязки:
			{ 
			  "success": true, 
			  "device_id": "a1b2c3d4-e5f6-7890-g1h2-i3j4k5l6m7n8",
			  "message": "Device bound and registered",
			  "cloud_url": "https://crcerror.ru"
			}
	4. Сайт завершает привязку:
		Обновляет статус устройства на "active":
			UPDATE devices SET status = 'active', bound_at = NOW() 
			WHERE device_id = 'a1b2c3d4-e5f6-7890-g1h2-i3j4k5l6m7n8'
		Показывает пользователю сообщение об успешной привязке
4.1.4. Рабочий режим с использованием Device ID и синхронизация состояния привязки
	1. Синхронизация состояния привязки при запуске ESP32:
		При каждом запуске (включении питания или перезагрузке) ESP32 выполняет синхронизацию:
			1. Загружает binding.json через BindingManager (источник истины для состояния привязки)
			2. Загружает cloud.json через StorageManager в systemState
			3. КРИТИЧЕСКИ ВАЖНО: Выполняет синхронизацию и reconciliation (согласование):
				Если bindingManager.isBound() == true:
					- Проверяет согласованность с systemState:
						* systemState.deviceBound должен быть true
						* systemState.deviceId должен совпадать с bindingManager.getUUID()
						* systemState.apiToken должен совпадать с bindingManager.getAPIToken()
					- При обнаружении несоответствия:
						* Логирует предупреждение с деталями несоответствия
						* Обновляет systemState из BindingManager (binding.json - источник истины)
						* Сохраняет исправленное состояние в cloud.json через storageManager.saveCloudSettings()
						* Логирует успешное согласование
				Если bindingManager.isBound() == false:
					- Проверяет, что systemState.deviceBound == false
					- При несоответствии обновляет systemState.deviceBound = false
					- Сохраняет исправленное состояние в cloud.json
			4. Логирует результат синхронизации:
				- "Binding states are consistent" если состояния согласованы
				- "Detected binding state mismatch - reconciling" при обнаружении несоответствия
				- "Reconciled binding state saved successfully" после успешного согласования
		Это гарантирует, что после перезагрузки или перепрошивки (LittleFS сохраняется):
			- systemState.deviceBound всегда соответствует реальному состоянию привязки
			- Отображение "Bound: Yes/No" в логах корректно
			- Конфликты между binding.json и cloud.json автоматически разрешаются
	2. Отправка данных с ESP32 на сайт:
		Каждые 60 секунд ESP32 отправляет данные (формат см. раздел 2.4.2).
			POST https://crcerror.ru/api/send-data.php
		В ответе сервер возвращает список команд ("commands"). ESP32 обязан:
			Выполнить каждую команду по типу (emergency_stop, start_program и т.д.)
			В следующем запросе передать список выполненных ID: "executed_commands": [42, 43]
		Сайт проверяет device_id и api_token перед приемом данных
	3. Загрузка программ с сайта на ESP32:
		Программы передаются только по инициативе сайта (автоматическая периодическая
		синхронизация программ на ESP32 не используется).
		При отправке программы сайт выполняет:
			POST http://<esp32_ip>/api/load-from-cloud
			{
			  "device_id": "a1b2c3d4-e5f6-7890-g1h2-i3j4k5l6m7n8",
			  "programs": [
				{
				  "name": "Сёмга холодного копчения",
				  "stages": [
					{
					  "stage_name": "Сушка",
					  "target_temp": 25,
					  "target_temp_device": 0,
					  "target_humidity": 70,
					  "duration_minutes": 240,
					  "hysteresis": 2,
					  "wait_for_temp": true,
					  "use_smoke_generator": false,
					  "ventilation_percent": 70,
					  "internal_fan_on": true,
					  "injection_fan_on": true,
					  "compressor_pwm": -1
					}
				  ]
				}
			  ]
			}
		ESP32 проверяет device_id, сохраняет программы в LittleFS и ОБЯЗАН отправить
		подтверждение на сервер:
			POST https://crcerror.ru/api/program-received.php
			{
			  "device_id": "a1b2c3d4-e5f6-7890-g1h2-i3j4k5l6m7n8",
			  "device_token": "<api_token>",
			  "received_programs": ["Сёмга холодного копчения"],
			  "status": "ok"
			}
		При ошибке сохранения ESP32 отправляет "status": "error" с полем "error_message".
	4. Подтверждение получения файлов (новое требование):
		Все файлы, передаваемые с сайта на ESP32, должны подтверждаться:
			- Программы (POST /api/load-from-cloud)
			- Настройки WiFi (POST /api/wifi)
			- Настройки облака (POST /api/cloud-settings)
			- OTA обновления (POST /update)
		ESP32 сохраняет файл в LittleFS и отправляет подтверждение с:
			- "status": "ok" при успешном сохранении
			- "status": "error" + "error_message" при ошибке
		Сайт считает файл переданным только после получения подтверждения
		При отсутствии подтверждения в течение 10 секунд сайт повторяет отправку (максимум 3 раза)
4.2. Обработка ошибок и восстановление после сбоев
4.2.1. Потеря связи с облаком
	1. Обнаружение потери связи:
		ESP32 пытается отправить данные каждые 60 секунд
		При 3 последовательных неудачных попытках:
		Система переходит в автономный режим
		На OLED отображается:
			! OFFLINE !

			Локальное управление:
			MENU - меню программ
			OK - запуск/стоп
	2. Восстановление связи:
		ESP32 пытается восстановить связь каждые 5 минут
		При успешном подключении:
		Отправляются накопленные данные (последние 10 записей, каждая с run_id)
		Синхронизируется текущее состояние
		На OLED отображается:
			! ВОССТАНОВЛЕНИЕ СВЯЗИ !
			Отправлено: 8 записей
4.2.2. Ошибка аутентификации (неверный Device ID или api_token)
	1. При получении данных с неверным device_id или невалидным api_token:
		Сайт проверяет существование устройства:
			SELECT id FROM devices WHERE device_id = ? AND status = 'active'
		При отсутствии результата или невалидном api_token возвращает HTTP 401 Unauthorized
		ESP32 логирует ошибку и отображает на OLED:
			! AUTH ERROR !

			Device ID не распознан.
			Проверьте привязку на сайте
	2. Восстановление после ошибки:
		Если причина — неверный api_token: ESP32 отображает ошибку на OLED и предлагает
		пользователю повторно привязать устройство через страницу /bind
		Если причина — неверный device_id: ESP32 пытается повторно отправить данные
		каждые 30 секунд (максимум 5 попыток), затем возвращается в режим AP
4.2.3. Аварийная остановка и уведомления
	1. При аварийной остановке (любой источник: кнопка, перегрев, команда с сайта):
		Все исполнительные механизмы отключаются немедленно (<100 мс)
		ESP32 отображает на OLED:
			! АВАРИЙНАЯ ОСТАНОВКА !
			<причина>
		ESP32 отправляет уведомление на сервер (через команду в следующем send-data.php
		либо немедленно POST /api/emergency-stop.php с device_id и device_token)
	2. Сервер при получении аварийной остановки:
		Устанавливает статус активного run = 'stopped'
		Отправляет email-уведомление владельцу устройства:
			Тема: «⚠️ АВАРИЙНАЯ ОСТАНОВКА — Smart Smoker»
			Содержание: устройство, причина остановки, время, ссылка на панель управления
		Логирует событие в системный журнал

9. ДОПОЛНИТЕЛЬНЫЕ ТРЕБОВАНИЯ (уточнения по результатам разработки)

9.1. Телеметрия привязана к запускам программ (run_id)
	Каждая запись телеметрии в таблице sensor_data должна содержать поле run_id.
	run_id — UUID, генерируемый ESP32 при старте программы (SystemState.currentRunId).
	При отправке данных (POST /api/send-data.php) ESP32 передаёт run_id в поле system.run_id.
	Сервер сохраняет run_id в sensor_data.run_id.
	Если устройство в режиме IDLE — run_id = NULL.
	Веб-интерфейс (view-device.php, dashboard.php) отображает телеметрию в контексте
	последних 10 запусков программ, группируя данные по run_id.
	Таблица sensor_data должна иметь индекс по run_id для эффективной выборки.

9.2. Хранение программ на ESP32 — отдельные файлы
	Каждая программа копчения хранится как отдельный JSON файл в LittleFS:
		/programs/program_{id}.json  — программы, полученные с сайта (по transfer_id)
		/programs/program_{name}.json — программы, созданные локально через веб-интерфейс ESP32
	Единый файл /programs.json не используется для хранения программ.
	StorageManager.saveProgram(int programId, json) — сохранение программы с сайта.
	StorageManager.saveProgram(SmokingProgram) — сохранение локальной программы.
		ВАЖНО: Локальные программы также сохраняются с префиксом "program_" в имени файла:
		Имя файла формируется как: /programs/program_{sanitized_name}.json
		где sanitized_name — имя программы в нижнем регистре с заменой пробелов на "_"
		Пример: программа "Сёмга холодного копчения" → /programs/program_semga_holodnogo_kopcheniya.json
		Поле "program_name" внутри JSON обязательно для всех программ (и локальных, и с сайта).
	При загрузке программ ESP32 читает все файлы из директории /programs/.
	ProgramIndex (/programs/index.json) хранит метаданные всех программ для быстрого листинга.

9.3. Ограничение Serial.print() логирования
	В production-режиме ESP32 выводит в Serial только:
		- Критические ошибки ([ERROR])
		- Предупреждения ([WARN])
		- Старт/стоп программ
		- Изменение состояния привязки
		- Аварийные остановки
	Запрещены в production: вывод каждого HTTP запроса, буферизации данных,
	retry очереди, периодической статистики (printSystemInfo не чаще 1 раза в 4 часа).
	Режим отладки (DEBUG_MODE) в SmartSmokerNetworkManager остаётся до завершения
	аппаратного тестирования.

10. ДОПОЛНЕНИЯ ПО РЕЗУЛЬТАТАМ РЕАЛИЗАЦИИ (2026-03)

10.1. OTA-обновление прошивки через облако (автоматическое)

10.1.1. Описание
	Реализована система автоматического OTA-обновления прошивки ESP32 через облачный сервер.
	Контроллер самостоятельно проверяет наличие новой прошивки и скачивает её без участия пользователя.

10.1.2. Таблица разделов (partitions.csv)
	Используется кастомная таблица разделов с двумя OTA-слотами:
		Name,   Type, SubType, Offset,   Size,     Flags
		nvs,    data, nvs,     0x9000,   0x5000,
		otadata,data, ota,     0xe000,   0x2000,
		app0,   app,  ota_0,   0x10000,  0x1E0000,
		app1,   app,  ota_1,   0x1F0000, 0x1E0000,
		spiffs, data, spiffs,  0x3D0000, 0x30000,
	Размер каждого OTA-слота: 1.875 МБ (достаточно для прошивки ~1.5 МБ).

10.1.3. Процесс OTA-обновления на ESP32 (AutoUpdateClient)
	1. Проверка обновлений: GET /api/check-update.php?device_id=...&current_version=...&api_token=...
	   Ответ сервера:
	   {
	     "update_available": true,
	     "version": "2.2.1",
	     "url": "https://crcerror.ru/api/download-firmware.php",
	     "size": 1501920,
	     "checksum": "fc7054b2..."
	   }
	2. Скачивание прошивки: GET /api/download-firmware.php?version=2.2.1&device_id=...&api_token=...
	   - Потоковая запись напрямую в Update.write() (без буферизации в RAM)
	   - Сброс watchdog (esp_task_wdt_reset()) каждые 4 КБ
	   - Таймаут HTTP: 30 секунд
	   - Retry-механизм: 3 попытки при ошибке
	3. Верификация: SHA256 скачанной прошивки сравнивается с checksum из ответа сервера
	4. Применение: Update.end() → ESP32 перезагружается с новой прошивкой
	5. Логирование: UpdateLogger сохраняет историю обновлений в LittleFS (/update_log.json)

10.1.4. Серверная часть OTA
	GET /api/check-update.php
		Параметры: device_id, current_version, api_token
		Аутентификация: проверка api_token устройства
		Логика: возвращает последнюю активную прошивку из таблицы firmware_updates
		Ответ: JSON с полями update_available, version, url, size, checksum

	GET /api/download-firmware.php
		Параметры: version (опционально), device_id, api_token
		Аутентификация: проверка api_token устройства
		Логика: отдаёт бинарный файл прошивки из папки /firmware/
		Заголовки: Content-Type: application/octet-stream, Content-Length: <размер>
		Логирование: запись в таблицу firmware_downloads

	Таблица firmware_updates (БД):
		- version (UNIQUE) — версия прошивки
		- filename — имя файла в папке /firmware/
		- file_path — полный путь к файлу
		- file_size — размер в байтах
		- checksum — SHA256 хеш
		- is_active — активна ли прошивка (только одна активна одновременно)
		- release_notes — описание изменений
		- is_required — обязательное обновление
		- min_version_required — минимальная версия для обновления

10.1.5. Управление прошивками через админ-панель
	Страница admin/firmware.php:

10.3. Система передачи программ на устройство (Pull-модель)

10.3.1. Описание
	Реализована pull-модель передачи программ: ESP32 сам скачивает программу с сервера
	по запросу, вместо того чтобы сервер пушил программу на ESP32.

10.3.2. Процесс передачи
	1. Пользователь нажимает "Отправить на устройство" на сайте
	2. Сайт создаёт запись в таблице program_transfer_queue со статусом 'pending'
	3. ESP32 при следующем опросе GET /api/get-state.php получает команду на скачивание
	4. ESP32 скачивает программу: GET /api/download-program.php?program_id=...&device_id=...&api_token=...
	5. ESP32 сохраняет программу в LittleFS и отправляет подтверждение:
	   POST /api/confirm-program-transfer.php
	   { "transfer_id": "...", "device_id": "...", "api_token": "...", "status": "confirmed" }
	6. Сайт обновляет статус в program_transfer_queue на 'confirmed'

10.3.3. Retry-механизм
	При ошибке передачи:
		- Статус в program_transfer_queue → 'failed'
		- Пользователь может нажать "Повторить" → POST /api/retry-program-transfer.php
		- Создаётся новая запись в очереди с previous_attempt_id
		- Максимум 3 попытки

10.3.4. Удаление программы с устройства
	POST /api/delete-program-from-device.php
		Параметры: program_id, device_id (числовой ID устройства)
		Логика: удаляет запись из device_programs, отправляет команду на ESP32

10.4. Привязка устройства (Pull-модель)

10.4.1. Описание
	Реализована pull-модель привязки: ESP32 сам инициирует привязку к сайту,
	используя логин/пароль пользователя. Сайт не пушит Device ID на ESP32.

10.4.2. Процесс привязки (новый, через сайт)
	1. Пользователь на сайте нажимает "Привязать" для устройства со статусом 'inactive'
	2. Сайт создаёт запрос на привязку: POST /api/bind-request.php
	   Ответ: { "request_id": "UUID", "status": "pending" }
	3. ESP32 периодически опрашивает: GET /api/bind-result.php?request_id=...&device_id=...
	4. Сайт обрабатывает запрос, генерирует api_token, обновляет статус устройства на 'active'
	5. ESP32 получает api_token и сохраняет в binding.json и cloud.json

10.4.3. Статусы устройств
	'pending'  — устройство создано, ожидает привязки
	'active'   — устройство привязано к пользователю
	'inactive' — устройство отвязано (unbound=0, но статус сброшен)
	Кнопка "Привязать" отображается ТОЛЬКО для устройств со статусом != 'active'

10.12. Подтверждение розжига дымогенератора

10.12.1. Описание проблемы
	На этапе копчения компрессор дымогенератора запускается автоматически, однако
	пользователь должен вручную поджечь щепу. Таймер этапа не должен стартовать
	до тех пор, пока дымогенератор не вышел на рабочий режим.

10.12.2. Новое состояние системы: WAITING_SMOKE_IGNITION
	Добавлено в перечисление SystemState::Mode (SystemState.h):
		IDLE, RUNNING, PAUSED, WAITING_SMOKE_IGNITION, EMERGENCY_STOP
	Состояние активируется при старте любого этапа с флагом useSmokeGenerator=true,
	после того как достигнута целевая температура (если waitForTemp=true).

10.12.3. Логика работы на ESP32 (ProgramManager.h)

	Вход в WAITING_SMOKE_IGNITION:
		- Запоминается базовая температура дыма (smokeBaselineTemp = tempSmoke)
		- Запускается компрессор дымогенератора (ШИМ по настройкам этапа или 70%)
		- Поддерживается температура камеры/продукта
		- Таймер этапа НЕ запускается
		- Фиксируется время входа (smokeIgnitionStartTime)

	Авто-детекция розжига:
		- Условие: tempSmoke - smokeBaselineTemp >= 10°C (SMOKE_DETECTION_DELTA)
		- Порог подобран экспериментально, может быть скорректирован

	Ручное подтверждение:
		- Кнопка OK на контроллере
		- Кнопка в веб-интерфейсе ESP32 (POST /api/smoke-ready)
		- Кнопка на сайте/PWA (POST /api/smoke-confirmed.php → команда smoke_confirmed)

	При подтверждении (авто или ручном):
		- mode → RUNNING
		- stepStartTime = текущее время (таймер этапа стартует с этого момента)

	Таймаут ожидания (10 минут, SMOKE_IGNITION_TIMEOUT_MS):
		- Если розжиг не подтверждён за 10 минут → переход в паузу (smokePauseActive=true)
		- mode → PAUSED
		- Температура поддерживается, компрессор продолжает работать
		- Уведомление пользователю каждые 10 минут (SMOKE_NOTIFY_INTERVAL_MS)

	Максимальное время паузы (30 минут, SMOKE_PAUSE_MAX_MS):
		- Если за 30 минут паузы розжиг не подтверждён → аварийная остановка
		- emergencyReason = "Дымогенератор не розжёгся в течение 30 минут"

	Переход к следующему этапу:
		- Флаги smokeIgnitionConfirmed, smokeBaselineTemp и т.д. сбрасываются
		- Каждый этап с useSmokeGenerator=true проходит процедуру заново

10.12.4. Новые поля в SystemState.h
	unsigned long smokeIgnitionStartTime   — millis() входа в WAITING_SMOKE_IGNITION
	float smokeBaselineTemp                — базовая температура дыма при старте
	bool smokeIgnitionConfirmed            — флаг подтверждения розжига
	unsigned long smokePauseStartTime      — millis() начала паузы
	bool smokePauseActive                  — флаг активной паузы
	unsigned long lastSmokeNotificationTime — время последнего уведомления

	Константы:
		SMOKE_DETECTION_DELTA    = 10.0°C
		SMOKE_IGNITION_TIMEOUT_MS = 10 мин
		SMOKE_PAUSE_MAX_MS        = 30 мин
		SMOKE_NOTIFY_INTERVAL_MS  = 10 мин

10.12.5. Новое поле в ProgramStep (ProgramStructures.h)

10.13. Автомат розжига дымогенератора (аппаратная интеграция)

10.13.1. Назначение
	Автомат розжига — внешнее устройство, которое по команде от ESP32
	выполняет розжиг газовой горелки дымогенератора и сообщает результат
	обратно на контроллер. Интеграция реализована через два цифровых пина.

10.13.2. Пины (pins.h)
	PIN_IGNITER_CMD    = GPIO27 (цифровой выход)
		Команда на запуск розжига. ESP32 подаёт импульс HIGH длительностью
		500 мс, затем возвращает в LOW.
	PIN_IGNITER_STATUS = GPIO36 (VP, цифровой вход, input-only)
		Ответный сигнал от автомата розжига. В норме — LOW (0).
		Переходы 0→1 несут информацию о результате розжига.

10.13.3. Протокол ответа автомата розжига
	Отсчёт начинается с момента первого перехода 0→1 на PIN_IGNITER_STATUS.
	Окно анализа: 5000 мс (IGNITER_SIGNAL_WINDOW).
	Подсчитывается количество переходов 0→1 за это окно:

		1 переход  → розжиг успешен (горелка зажглась и отработала)
		2 перехода → газ закончился в баллоне
		3 перехода → газ замёрз

	Если за 30 секунд после команды первый переход не поступил — TIMEOUT.
	Если переходов больше 3 — ERROR (неожиданный сигнал).

10.13.4. Реализация (IgniterManager.h)
	Класс IgniterManager инкапсулирует всю логику:
		begin()          — инициализация пинов (вызов в setup())
		startIgnition()  — запуск процедуры розжига
		update()         — вызывать в каждой итерации loop()
		isDone()         — true когда результат готов
		getResult()      — возвращает IgniterResult enum
		reset()          — сброс в IDLE после обработки результата

	Состояния (IgniterState):
		IDLE      → CMD_SENT → COUNTING → DONE

	Результаты (IgniterResult):
		SUCCESS    (1) — розжиг успешен
		NO_GAS     (2) — газ закончился
		GAS_FROZEN (3) — газ замёрз
		TIMEOUT    (4) — нет ответа
		ERROR      (5) — неожиданный сигнал

10.13.5. Интеграция с ProgramManager
	Автомат розжига используется в связке с процедурой WAITING_SMOKE_IGNITION
	(раздел 10.12). После подтверждения розжига пользователем (вручную или
	через кнопку OK / веб-интерфейс) контроллер может дополнительно
	запросить аппаратное подтверждение от автомата розжига через
	IgniterManager::startIgnition().

	При результате NO_GAS или GAS_FROZEN:
		- Система остаётся в WAITING_SMOKE_IGNITION
		- На дисплее и в веб-интерфейсе отображается причина неудачи
		- Пользователю отправляется push-уведомление с описанием проблемы

10.13.6. Константы (constants.h)
	IGNITER_SIGNAL_WINDOW    = 5000 мс  — окно анализа сигналов
	IGNITER_CMD_PULSE_MS     = 500 мс   — длительность командного импульса
	IGNITER_RESULT_SUCCESS   = 1
	IGNITER_RESULT_NO_GAS    = 2
	IGNITER_RESULT_GAS_FROZEN = 3
